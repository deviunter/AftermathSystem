# AFTERMATH SYSTEM - СИСТЕМА ПОСЛЕДСТВИЙ
Как и система инвентаря, система последствий это духовный наследник системы последствий After Darkness, версия раньше использовала в качестве основоного класса
Actor и была полностью реализована на Blueprints, эта же система полностью находится в C++, и на этот раз устроена более удобно используя UWorldSubsystem.
О такой системе я задумался, когда поиграл в Death Stranding - мне стало интересно - а как игра сохраняет созданные обьекты и из этого родилась Aftermath System.
А с этой версии она использует FInstancedStruct в качестве универсального контейнера данных об сохраняемых акторах.

# КАК РАБОТАЕТ
Система подразделяется на несколько задач, она не только контейнер данных для сохранения об акторах, но и глобальный трекер. Она отслеживает перемещения игрока по миру
через кастомные volume, передавая это через делегаты всем, кому требуется знать - где находится игрок, например квестовой системе, для задач по типу:
"Доберитесь до места назначения", "Покиньте область" или что-то подобное.

**СОХРАНЕНИЕ ПАРАМЕТРОВ АКТОРОВ**
Несмотря на то, что система расчитана на сохранение акторов и их дочерних классов их можно использовать и для других классов - например единоразового появления каких-то виджетов.
Сохранения подразделяются на несколько типов.
1. SimplePresetActorsInfo - массив данных для "простых" акторов, например записок, кому достаточно лишь галочки или банального упоминания в системе. Важно, что в эту категорию могут быть отнесены акторы, которые были размещены в редакторе
2. PresetActorsInfo - массив данных о "сложных" акторах, которые были размещены в редакторе.
3. GeneratedActorsInfo - массив данных о акторах, которые были созданы в рантайме, во время игровой сессии, поскольку напрямую привязать их к какому то саблевелу нельзя, эта система имитируется с помощью регионального трекера.
4. StashParameters и WeaponParameters - данные сохранения об оружии игрока и схронах в мире.

Однако, один и тот же класс может быть как размещён в редакторе так и заспавнен в рантайме. Для определения этого используется Enumerator EPresenceType, который обновляется в акторе в момент его спавна, так как по дефолту он "Editor", а при спавне
выставляется как "Generated". На мой взгляд это костыльный метод, но лучшего решения я пока не нашёл.
Ещё одной проблемой является то, что ID акторов в системе составляется из их SafePathName - это уникальный для каждого экземпляра класса параметр, у меня возникала бы проблема, что новый актор не знает ID старого актора и система бы просто не работала
корректно. Поэтому это было решено тем, что в момент спавна актора через Aftermath, если актор уже в системе, происходит подмена ID, на SafePathName нового актора. Таким образом эта проблема тоже решается.

**РЕГИОНАЛЬНЫЕ ПАРАМЕТРЫ**
Игра, ввиду её сюжета поделена на отдельные регионы и локации. Поэтому этой системе необходимо отслеживать глобально нахождение игрока в мире, для внешних систем и для собственной системы генерации акторов, так как в зависимости от параметров актора
он может быть заспавнен как при появлении игрока в регионе так и ждать появления его в локации. Тоесть сама архитектура мира устроена так: `Persistent Level > Region (RegionVolume) Sublevel > Location (LocationVolume) Sublevel.
